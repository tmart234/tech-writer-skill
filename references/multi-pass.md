# Multi-Pass Architecture: full pass instructions

> Load this file before starting Pass 0. SKILL.md carries the five-pass skeleton; this file carries the detailed sub-tasks for each pass. Pass 0 (evidence) and Pass 0.5 (stakes) are the two pre-draft passes most commonly skipped, and the ones that separate real writing from fluent-but-empty prose.
>
> Sub-tasks marked *(voice profile only)* run only when a `voice/` folder is present. Without one, skip them.
>
> **Reviewing an existing draft?** Skip Pass 0 and Pass 0.5 as production steps and start at Pass 2, then Pass 3. The draft still has to clear the Pass 0 and Pass 0.5 bar: if Pass 3 surfaces a claim with no evidence or a flaw with no named stakes, drop back, do the missing Pass 0 or Pass 0.5 work, and re-draft the affected sections.

0. **Evidence-gathering pass (before any drafting).** Sub-tasks, in order:
   - **Value-to-whom sentence first.** Write one sentence: *"This post is valuable to [specific reader] because [specific change in how they think or act]."* (See `craft-of-writing.md` §2.) If you can't fill in either bracket, stop. There's no post yet. The bracket-fill anchors every other decision: which evidence is doing real work, which counter-evidence the reader will demand, which content type fits. Name the *primary* reader. **Past-self trick when the blank is hard to fill:** name a version of yourself at the specific moment you needed this answer — when you were debugging the bug, evaluating the tool, reading the spec for the first time, picking between two designs. If you can name the moment in your own history when this post would have changed what you did, the value-to-whom sentence has bite. Future-self works too for posts that anticipate a known upcoming decision. The generator fails the same way the rule does: if you can't name the moment, you don't yet have a post. *(voice profile only)* If a voice profile defines an audience model (`voice/profile.md`), name the primary audience from it; the post can land with several readers, but it's pitched at one.
   - **Then list the concrete stuff** the post will lean on: CVEs with IDs, specific code paths with filenames and line numbers, command outputs, measurements, packet captures, incidents with dates, citations to primary sources (upstream advisories, standards documents, original research papers), field observations. For opinion posts, also list the evidence that would *disprove* the thesis. If you can't state that, you don't have a thesis yet, you have a vibe. Rule of thumb: if the piece makes N distinct technical claims, you need ≥N evidence items, not a general "I've seen this a lot." Fluent, anti-pattern-free prose with no evidence in it is the failure mode this pass exists to prevent. *(voice profile only)* If a voice profile defines beat-specific evidence types (`voice/beat-notes.md`), fold those in the same way.
   - *(voice profile only)* **Pick the post-level register.** If a voice profile defines register theory (`voice/toolkit.md`), pick the post-level register: the profile's measured default, or its louder earned exception. Reach for the louder register only when the body of evidence assembled above is overwhelming enough to back the volume across the entire post. Write the choice down explicitly in the Pass 0 notes ("Register: [choice], backed by [evidence items 3, 7, 11]"). If you can't name the evidence that backs the louder register, default back to measured. This commitment is to the post-level register only; sentence-level register decisions are made in the Draft pass and validated in Pass 3. Without a voice profile there is no register dial: match tone to the strength of the evidence and move on.
   - *(voice profile only)* **Pick your influences.** If a voice profile ships a toolkit (`voice/toolkit.md`), open it and pick two or three writers' moves to steal from for this specific post: one structural move, one rhetorical-register move, one audience-anchor move. Write the picks down in the Pass 0 notes ("Influences: [structural] + [register] + [audience-anchor]"). The persona in `voice/profile.md` is the floor; the toolkit picks sharpen this specific post on top of it. Don't pick more than three; the voice splinters.
   - **Pick the method of development** (for posts longer than ~1,200 words). The content type from SKILL.md sets *what* the post is; the method of development sets *how the body organizes its evidence inside that type* — the two are orthogonal. Load `methods-of-development.md`, pick a primary method (and any secondaries), and write the picks into the Pass 0 notes ("Primary: comparison, part-by-part. Secondary: order-of-importance, decreasing."). If you can't pick, the body is still in discovery-path mode and the draft will inherit that shape. Skip for short posts — they carry linearly on the salient idea.

**Pass 0.5: Stakes pass (the generator, after evidence, before drafting).** This pass sits between Pass 0 and the Draft pass. It is where the post becomes a post and not a CVE write-up with adjectives. The Pass 0 evidence is the raw material; this pass turns it into stakes the reader can feel. Run it for every post, not just opinion and essay. Three sub-tasks, in order, and the output is a written stakes ledger that you cite in the Draft pass the way you cite CVEs.

   - **Name the loser. Concretely.** Who is worse off because this thing is broken, and what does "worse off" look like at the level of a person, a shift, an institution, or a dated incident? The discipline is cause → effect → who gets hurt, and the third clause is the one writers skip. *Not* "users are impacted" → *"the customer whose session token the XSS exfiltrated."* *Not* "the org is exposed" → *"the on-call engineer at 2 a.m. deciding whether to take the service down or ride it out."* *Not* "the outage was severe" → *"the 7,000 merchants who couldn't process payments during the Black Friday window."* Name at least one. Name three if you can. If you can't name a loser, you don't have a stakes ledger and the post will read inert no matter how clean the prose is.

   - **Name the winner.** Whose incentives produce this failure mode, and who is paid (in money, in time saved, in liability avoided, in promotions) for the system staying broken? This is the move that turns analysis into pointing. Name the role, the institution, or the named individual where you can. *Not* "vendors don't fix this" → *"the engineer who copy-pasted the vulnerable dependency in 2019 and the manager who decided re-testing was more expensive than shipping."* *Not* "the platform was slow to respond" → *"the security team with 300 open tickets and no authority to block a release."* If naming a specific person crosses a line you don't want to cross, name the role with enough specificity that the reader pictures one person, not a category. The winner doesn't have to be malicious. Most of the time they're rational actors responding to broken incentives, and saying so is more damning than calling them villains.

   - **Name the incentive that connects them.** What is the mechanism (the broken process, the misaligned ownership, the liability allocation, the standard that lets a vendor ship stale code) that lets the winner keep winning while the loser keeps losing? This is the seed of the abstraction or prescription move in the post's body. *"The dependency is three layers deep in the build, and nobody owns transitive updates."* *"The bug-bounty program pays less than the cost of triaging a report, so reports get closed as informational."* If you can't name a mechanism, the post will land as outrage rather than analysis. See `craft-of-writing.md` §6 on problem-in-the-world vs. problem-in-the-reader.

   **The output is a stakes ledger**, three to six bullets that name losers, winners, and the incentive that connects them. You cite it in the Draft pass the way you cite CVEs. A draft that has Pass 0 evidence but no Pass 0.5 ledger will pass the anti-pattern scan and still read inert, because earned is a permission, not an instruction. The ledger is the instruction. *(voice profile only)* If the voice profile defines sentence-level register punctuation, every such landing in the draft draws from this ledger and names a specific loser, winner, or incident from it.

1. **Draft pass.** Generate content following persona, content type, reader level, and structural rules. Cite the evidence items from Pass 0 inline as you go. Cite the stakes ledger from Pass 0.5 the same way: every section should be reaching toward a named loser or winner, not gesturing at "users" or "the industry." Don't leave "[TODO: add CVE]" breadcrumbs; if it's not cited, cut the claim. *(voice profile only)* When a paragraph has done enough work to earn a tactical register landing (see `voice/toolkit.md`), swing, and name someone from the ledger when you do.
2. **Anti-pattern pass.** Run the three sub-pass anti-pattern scan against `anti-patterns.md` (the word-level, phrase-and-structure, and smell-test-and-security sub-passes), then `../scripts/check-draft.sh` for the countable caps. *(voice profile only)* Also run the beat-specific anti-patterns in `voice/beat-notes.md`.
3. **Validation pass.** Work through the checklist below in order. Each item is a distinct check; run them one at a time rather than blurring them into a single read.

   1. **Evidence backing.** Confirm every technical claim is backed by the evidence assembled in Pass 0.
   2. **Sentence-length variance.** Check it; see `anti-patterns.md` §4b.
   3. **Heading hierarchy.** Verify it: one H1, no skipped levels, every heading a promise the section keeps.
   4. **Front-matter completeness.** Validate against the target platform's requirements.
   5. **Restated points.** Scan for them: when two sentences make the same claim, cut the weaker one.
   6. **Quality rubric.** Run it (see SKILL.md, *Quality rubric*). Below threshold triggers a rewrite pass.
   7. **Reader-level check.** Does every term and acronym either belong at the post's declared reader level or get defined inline? Walk through the post in order and flag the first time a term is used without context. If the reader could hit that term and not know what it is, fix it.
   8. *(voice profile only)* **Multi-audience clarity test.** Defined in `voice/profile.md`: walk the post once for each audience the profile names and ask what each one takes away; no audience should be actively excluded by jargon, register, or framing.
   9. *(voice profile only)* **Tactical register-landing check.** Defined in `voice/toolkit.md`: for each sentence doing sharper-register work inside an otherwise measured post, confirm both the local-evidence and named-stakes criteria.
   10. **Stakes-ledger trace.** For each bullet in the Pass 0.5 ledger, find where it landed in the post. A ledger bullet that didn't make it into the prose is either a missing landing (add it) or a stake that wasn't really doing work (cut it from the ledger and move on).
   11. **Consequence-anchor check.** A final sweep: for each technical flaw described, did the post reach the real-world consequence with names attached (cause → effect → who gets hurt)? If a flaw is named without a downstream loser, either Pass 0.5 didn't go deep enough or the draft skipped the ledger. Fix the draft, or go back to Pass 0.5 if the ledger itself was thin.
   12. **Plain-language pass.** Scan for four-syllable-plus words that aren't technical terms. For each one, ask if a shorter word does the job. If yes, swap.
   13. **Juxtaposition-and-uplift check.** Scan for the antithesis rhythm (`anti-patterns.md` §4q). If most key sentences in a section pivot on a contrast, break the run. Then check the negative-to-positive arc (§4r): if the closer's optimism would survive a worse result unchanged, it is structural rather than earned, so cut it.
   14. **Template scan.** Walk the draft sentence by sentence and ask of each one: would this sentence survive find-and-replace on its nouns? If you can swap one technology for another, or one domain for another, and the sentence still reads as true and complete, it's a template, not a claim. A real claim breaks under the swap because it's load-bearing on the specific thing. This catches the named shape patterns (the prophecy closer, `anti-patterns.md` §4p; the contrastive reframe, §3f; the rule of three, §4a) and the ones the skill hasn't named yet. For each template sentence: cut it, or back-fill the specific evidence, name, number, code path, or date that makes it break under the swap.
   15. **Salient-idea test.** If the reader will remember exactly one sentence from this post, which sentence will it be? If the answer is "I don't know" or "the closing paragraph," the post lacks a salient idea. Sharpen one before publishing. (See `how-to-speak.md` §4.)
   16. **Explain-vs-change test.** (McEnerney, see `craft-of-writing.md` §8.) For each paragraph, ask *what does the reader believe or do differently after reading this that they didn't before?* If the answer is "they now know what I knew," the paragraph is explaining; tighten it to the reader-side change or cut.
   17. **Read-aloud whole-post pass.** Read the entire post aloud, or run it through a text-to-speech reader (Natural Reader, the macOS `say` command, your phone's accessibility tools). Different from the sentence-level voice test in `writing-rules.md`: that test catches *register* (does this sound like a textbook or a colleague?); this one catches *rhythm, pacing, and flow* across the whole document — the problems that exist between sentences, not inside them. Signals to listen for: places you trip over the wording (rewrite), sentences that feel out-of-breath long (split), paragraphs that drone in metronomic rhythm (vary sentence length, see `anti-patterns.md` §4b), transitions that don't land aurally (the eye forgives jumps the ear doesn't), word choices that sound off in your own voice (replace with how you'd actually say it). TTS catches most of the same issues without forcing you to read aloud yourself; use it for routine posts and read aloud for the ones that matter most.
   18. **Open-ended reviewer pass** (optional but recommended for posts you care about). Before publishing, share the draft with two readers from different positions: one stakeholder (a colleague working on the same topic, a co-author of the underlying research, an editor who knows the beat) and one target-audience reader (someone at the post's declared reader level who hasn't seen the work). **Don't explain your intent up front.** Hand over the draft as-is and ask open-ended questions only: *"What did you get out of it?"* / *"What should change?"* / *"What did you expect that wasn't there?"* When a stranger encounters the post on the open web, they get no context from you; the draft has to stand on its own. The test is whether each reviewer's takeaways match the Pass 0 value-to-whom sentence. Mismatch is a signal to fix the draft, not the reviewer; if the reviewer needed context to "get it," strangers will too.
   19. **Structural restructuring check.** (See `craft-of-writing.md` §1.) Does the post follow your discovery path ("first I tried X, then Y, then Z") instead of the reader's optimal path? If so, restructure to deliver the salient claim first.
