# Multi-Pass Architecture: full pass instructions

> Load this file before starting Pass 0. SKILL.md carries the five-pass skeleton; this file carries the detailed sub-tasks for each pass. Pass 0 (evidence) and Pass 0.5 (stakes) are the two pre-draft passes most commonly skipped, and the ones that separate real writing from fluent-but-empty prose.
>
> Steps marked *(voice profile only)* run only when a `voice/` folder is present. Without one, skip them.

0. **Evidence-gathering pass (before any drafting).** Sub-tasks, in order:
   - **Value-to-whom sentence first.** Write one sentence: *"This post is valuable to [specific reader] because [specific change in how they think or act]."* (See `craft-of-writing.md` §2.) If you can't fill in either bracket, stop. There's no post yet. The bracket-fill anchors every other decision: which evidence is doing real work, which counter-evidence the reader will demand, which content type fits. Name the *primary* reader. *(voice profile only)* If a voice profile defines an audience model (`voice/profile.md`), name the primary audience from it; the post can land with several readers, but it's pitched at one.
   - **Then list the concrete stuff** the post will lean on: CVEs with IDs, specific code paths with filenames and line numbers, command outputs, measurements, packet captures, incidents with dates, citations to primary sources (upstream advisories, standards documents, original research papers), field observations. For opinion posts, also list the evidence that would *disprove* the thesis. If you can't state that, you don't have a thesis yet, you have a vibe. Rule of thumb: if the piece makes N distinct technical claims, you need ≥N evidence items, not a general "I've seen this a lot." Fluent, anti-pattern-free prose with no evidence in it is the failure mode this pass exists to prevent. *(voice profile only)* If a voice profile defines beat-specific evidence types (`voice/beat-notes.md`), fold those in the same way.
   - *(voice profile only)* **Pick the post-level register.** If a voice profile defines register theory (`voice/toolkit.md`), pick the post-level register: the profile's measured default, or its louder earned exception. Reach for the louder register only when the body of evidence assembled above is overwhelming enough to back the volume across the entire post. Write the choice down explicitly in the Pass 0 notes ("Register: [choice], backed by [evidence items 3, 7, 11]"). If you can't name the evidence that backs the louder register, default back to measured. This commitment is to the post-level register only; sentence-level register decisions are made in the Draft pass and validated in Pass 3. Without a voice profile there is no register dial: match tone to the strength of the evidence and move on.
   - *(voice profile only)* **Pick your influences.** If a voice profile ships a toolkit (`voice/toolkit.md`), open it and pick two or three writers' moves to steal from for this specific post: one structural move, one rhetorical-register move, one audience-anchor move. Write the picks down in the Pass 0 notes ("Influences: [structural] + [register] + [audience-anchor]"). The persona in `voice/profile.md` is the floor; the toolkit picks sharpen this specific post on top of it. Don't pick more than three; the voice splinters.

**Pass 0.5: Stakes pass (the generator, after evidence, before drafting).** This pass sits between Pass 0 and the Draft pass. It is where the post becomes a post and not a CVE write-up with adjectives. The Pass 0 evidence is the raw material; this pass turns it into stakes the reader can feel. Run it for every post, not just opinion and essay. Three sub-tasks, in order, and the output is a written stakes ledger that you cite in the Draft pass the way you cite CVEs.

   - **Name the loser. Concretely.** Who is worse off because this thing is broken, and what does "worse off" look like at the level of a person, a shift, an institution, or a dated incident? The discipline is cause → effect → who gets hurt, and the third clause is the one writers skip. *Not* "users are impacted" → *"the customer whose session token the XSS exfiltrated."* *Not* "the org is exposed" → *"the on-call engineer at 2 a.m. deciding whether to take the service down or ride it out."* *Not* "the outage was severe" → *"the 7,000 merchants who couldn't process payments during the Black Friday window."* Name at least one. Name three if you can. If you can't name a loser, you don't have a stakes ledger and the post will read inert no matter how clean the prose is.

   - **Name the winner.** Whose incentives produce this failure mode, and who is paid (in money, in time saved, in liability avoided, in promotions) for the system staying broken? This is the move that turns analysis into pointing. Name the role, the institution, or the named individual where you can. *Not* "vendors don't fix this" → *"the engineer who copy-pasted the vulnerable dependency in 2019 and the manager who decided re-testing was more expensive than shipping."* *Not* "the platform was slow to respond" → *"the security team with 300 open tickets and no authority to block a release."* If naming a specific person crosses a line you don't want to cross, name the role with enough specificity that the reader pictures one person, not a category. The winner doesn't have to be malicious. Most of the time they're rational actors responding to broken incentives, and saying so is more damning than calling them villains.

   - **Name the incentive that connects them.** What is the mechanism (the broken process, the misaligned ownership, the liability allocation, the standard that lets a vendor ship stale code) that lets the winner keep winning while the loser keeps losing? This is the seed of the abstraction or prescription move in the post's body. *"The dependency is three layers deep in the build, and nobody owns transitive updates."* *"The bug-bounty program pays less than the cost of triaging a report, so reports get closed as informational."* If you can't name a mechanism, the post will land as outrage rather than analysis. See `craft-of-writing.md` §6 on problem-in-the-world vs. problem-in-the-reader.

   **The output is a stakes ledger**, three to six bullets that name losers, winners, and the incentive that connects them. You cite it in the Draft pass the way you cite CVEs. A draft that has Pass 0 evidence but no Pass 0.5 ledger will pass the anti-pattern scan and still read inert, because earned is a permission, not an instruction. The ledger is the instruction. *(voice profile only)* If the voice profile defines sentence-level register punctuation, every such landing in the draft draws from this ledger and names a specific loser, winner, or incident from it.

1. **Draft pass.** Generate content following persona, content type (Step 1), reader level (Step 2), and structural rules. Cite the evidence items from Pass 0 inline as you go. Cite the stakes ledger from Pass 0.5 the same way: every section should be reaching toward a named loser or winner, not gesturing at "users" or "the industry." Don't leave "[TODO: add CVE]" breadcrumbs; if it's not cited, cut the claim. *(voice profile only)* When a paragraph has done enough work to earn a tactical register landing (see `voice/toolkit.md`), swing, and name someone from the ledger when you do.
2. **Anti-pattern pass.** Run the three-pass scan from Step 4 against `anti-patterns.md`, then `../scripts/check-draft.sh` for the countable caps. *(voice profile only)* Also run the beat-specific anti-patterns in `voice/beat-notes.md`.
3. **Validation pass.** Confirm every technical claim is backed by the evidence assembled in Pass 0. Check sentence-length variance. Verify heading hierarchy. Validate front-matter completeness. Scan for restated points: when two sentences make the same claim, cut the weaker one. Run the quality rubric. **Run the reader-level check:** does every term and acronym either belong at the post's declared level (Step 2) or get defined inline? Walk through the post in order and flag the first time a term is used without context. If the reader could hit that term and not know what it is, fix it. *(voice profile only)* **Run the multi-audience clarity test** defined in `voice/profile.md`: walk the post once for each audience the profile names and ask what each one takes away; no audience should be actively excluded by jargon, register, or framing. *(voice profile only)* **Run the tactical register-landing check** defined in `voice/toolkit.md`: for each sentence doing sharper-register work inside an otherwise measured post, confirm both the local-evidence and named-stakes criteria. **Run the stakes-ledger trace.** For each bullet in the Pass 0.5 ledger, find where it landed in the post. A ledger bullet that didn't make it into the prose is either a missing landing (add it) or a stake that wasn't really doing work (cut it from the ledger and move on). **Run the consequence-anchor check** as a final sweep: for each technical flaw described, did the post reach the real-world consequence with names attached (cause → effect → who gets hurt)? If a flaw is named without a downstream loser, either Pass 0.5 didn't go deep enough or the draft skipped the ledger. Fix the draft, or go back to Pass 0.5 if the ledger itself was thin. **Run the plain-language pass:** scan for four-syllable-plus words that aren't technical terms. For each one, ask if a shorter word does the job. If yes, swap. **Run the juxtaposition-and-uplift check:** scan for the antithesis rhythm (`anti-patterns.md` §4q). If most key sentences in a section pivot on a contrast, break the run. Then check the negative-to-positive arc (§4r): if the closer's optimism would survive a worse result unchanged, it is structural rather than earned, so cut it. **Run the template scan:** walk the draft sentence by sentence and ask of each one: would this sentence survive find-and-replace on its nouns? If you can swap one technology for another, or one domain for another, and the sentence still reads as true and complete, it's a template, not a claim. A real claim breaks under the swap because it's load-bearing on the specific thing. This catches the named shape patterns (the prophecy closer, `anti-patterns.md` §4p; the contrastive reframe, §3f; the rule of three, §4a) and the ones the skill hasn't named yet. For each template sentence: cut it, or back-fill the specific evidence, name, number, code path, or date that makes it break under the swap. **Run the salient-idea test:** if the reader will remember exactly one sentence from this post, which sentence will it be? If the answer is "I don't know" or "the closing paragraph," the post lacks a salient idea. Sharpen one before publishing. (See `how-to-speak.md` §4.) **Run the explain-vs-change test** (McEnerney, see `craft-of-writing.md` §8): for each paragraph, ask *what does the reader believe or do differently after reading this that they didn't before?* If the answer is "they now know what I knew," the paragraph is explaining; tighten it to the reader-side change or cut. **Run the structural restructuring check** (see `craft-of-writing.md` §1): does the post follow your discovery path ("first I tried X, then Y, then Z") instead of the reader's optimal path? If so, restructure to deliver the salient claim first.
